# Архитектура и разработка `virtualization-lib`

Этот документ — карта проекта: как устроена структура, как работает сам компонент, какими инструментами и почему собирается и тестируется библиотека. Читается сверху вниз.

---

## 1. Что это за проект

Это **npm-библиотека** из одного React-компонента — `VirtualScroller`. Библиотека, в отличие от приложения:

- не запускается сама по себе — её **бандлит проект-потребитель**;
- поэтому она **не тащит React внутрь себя** (React — `peerDependency`, живёт у потребителя);
- поставляет наружу три вещи: **JS (ESM)**, **типы (`.d.ts`)** и **корректный манифест `package.json`**, который объясняет бандлеру потребителя, где что лежит.

Всё остальное в репозитории (стенд, конфиги) — это «леса» для разработки, они **не попадают в публикуемый пакет**.

## 2. Структура репозитория

```
.
├── src/
│   ├── index.ts                    # публичный API — единственная точка входа пакета
│   ├── VirtualScroller.tsx         # сам компонент (это и есть библиотека)
│   └── VirtualScroller.stories.tsx # Ladle-stories — стенд для ручного теста в браузере
├── dist/                           # результат сборки (в .gitignore, публикуется в npm)
│   ├── index.js                    # ESM-бандл
│   ├── index.js.map                # sourcemap
│   └── index.d.ts                  # типы
├── tsup.config.ts                  # конфиг сборки библиотеки
├── tsconfig.json                   # один tsconfig — для проверки типов и редактора
├── eslint.config.js                # flat-config ESLint
├── package.json                    # манифест пакета
├── README.md                       # витрина: установка, API, примеры (для пользователей)
└── ARCHITECTURE.md                 # этот файл (для разработчика либы)
```

**Ключевая граница:** `src/` содержит только код библиотеки (`index.ts`, `VirtualScroller.tsx`) и стенд (`*.stories.tsx`). Никакого демо-приложения, API-слоя или моков в `src/` больше нет — они смешивали «либу» и «песочницу» и загрязняли граф сборки.

Что публикуется в npm определяется полем `"files": ["dist"]` — уедет **только `dist/`** (+ `README`, `LICENSE`, `package.json`, которые npm добавляет всегда). Проверить это: `npm pack --dry-run`.

## 3. Логика компонента `VirtualScroller`

Виртуализация = **рендерим не весь список, а только видимое окно** + небольшой запас сверху/снизу. Остальное «занимаем местом» двумя пустыми `<div>`, чтобы полоса прокрутки была корректной длины.

Схема DOM:

```
┌─ контейнер (height = visibleItemsCount * itemHeight, overflow-y: auto) ─┐
│  <div style="height: topOffsetPx" />        ← пустышка сверху           │
│  [ отрендеренные элементы окна + padding ]   ← реальные ноды            │
│  <div style="height: bottomOffsetPx" />     ← пустышка снизу            │
└──────────────────────────────────────────────────────────────────────┘
```

Как считаются границы окна:

1. **`overviewStartIndex`** — первый видимый элемент. Обновляется в `onScroll`:
   `Math.floor(scrollTop / itemHeight)` — сколько элементов «уехало» вверх.
2. **`paddingStartIndex` / `paddingEndIndex`** — окно, расширенное на `paddingItemsCount` в обе стороны (буфер, чтобы при быстром скролле не мелькала пустота), с зажимом в границы `[0, itemsCount]`.
3. **`topOffsetPx`** = `paddingStartIndex * itemHeight` — высота верхней пустышки.
4. **`bottomOffsetPx`** = `realHeight - paddingEndIndex * itemHeight` — высота нижней пустышки, где `realHeight = itemHeight * itemsCount`.
5. Рендерим `items.slice(paddingStartIndex, paddingEndIndex).map(renderItem)`.

`useEffect` на `startIndexDefault` один раз проставляет `scrollTop`, чтобы список открылся на нужном элементе.

**Что было исправлено при рефакторинге:** раньше проп `items` копировался в локальный стейт `innerItems` и синхронизировался через `useEffect`. Это дублировало источник истины и давало лишний рендер. Теперь срез берётся напрямую из `items` — стейт остался только один и по делу (`overviewStartIndex`).

**Ограничения дизайна (осознанные):** высота элемента фиксированная (`itemHeight`), размер окна задаётся числом (`visibleItemsCount`), а не измеряется от контейнера. Это делает либу простой и быстрой, но не подойдёт для элементов переменной высоты — там нужна динамическая измерялка (как в `@tanstack/virtual`).

## 4. Инструменты и зачем каждый

| Инструмент | Роль | Почему он |
| --- | --- | --- |
| **tsup** | сборка библиотеки | На esbuild, почти без конфига. Одной командой даёт ESM + `.d.ts`. Для либы из одного модуля — самый короткий путь. |
| **TypeScript (tsc)** | проверка типов | Эмитом занимается tsup; `tsc --noEmit` — только валидация типов (`npm run typecheck`) и подсказки в редакторе. |
| **Ladle** | дев-стенд в браузере | Лёгкая Vite-native замена Storybook. Каждое состояние компонента = «story», компонент разрабатывается в изоляции, а не внутри ручного приложения. |
| **ESLint** | линт | Flat-config, TS + правила react-hooks. |

### Как собирается либа (`tsup.config.ts`)

- `entry: ["src/index.ts"]` — сборка идёт от публичного API.
- `format: ["esm"]` — только ESM (современные бандлеры это едят; CJS/UMD для React-либы сегодня не нужен).
- `dts: true` — генерит `dist/index.d.ts`.
- `external: ["react", "react-dom", "react/jsx-runtime"]` — **самое важное**: React не бандлится (он peer), и `react/jsx-runtime` тоже обязан быть external, иначе рантайм JSX уедет в пакет и у пользователя будет два React.
- `clean: true`, `sourcemap: true`, `treeshake: true`.

### Манифест (`package.json`) — что важно

- `"exports"` — **единый источник истины** для резолвера. Внутри условие `"types"` идёт **первым** (этого требуют `moduleResolution: bundler/node16`).
- `"types": "./dist/index.d.ts"` — путь к типам совпадает с точкой входа.
- `"sideEffects": false` — разрешает потребителю tree-shaking.
- `"peerDependencies": { "react": ">=18" }` — диапазон, а не `^19`, чтобы не отсекать React 18.
- `"files": ["dist"]` — в пакет уедет только сборка.
- `"prepublishOnly": "npm run build"` — защита от публикации без свежей сборки.

### Проверка пакета перед публикацией (полезные привычки)

```bash
npm run build              # собрать
npm pack --dry-run         # посмотреть, что реально уедет в тарбол
npx publint                # линт корректности exports/manifest
npx @arethetypeswrong/cli --pack   # проверка резолвинга типов во всех режимах
```

## 5. Тестирование

### Ручное — Ladle

```bash
npm run dev      # ladle serve — открывает стенд со stories
```

Стенды описаны в `src/VirtualScroller.stories.tsx`: базовый список, пустой, один элемент, 100k элементов (производительность), кастомный `startIndexDefault`, кастомные размеры. Добавлять состояние = добавить экспорт-функцию `Story`.

### Юнит-тесты — Vitest + RTL (настроены)

```bash
npm test            # прогон один раз
npm run test:watch  # watch-режим
npm run test:coverage
```

- Раннер — **Vitest** (не Jest): ESM-нативный, TS/TSX без конфига, тот же esbuild-трансформ, что у tsup. Матчеры jest-dom подключаются в `test/setup.ts` через `expect.extend` (глобалы vitest выключены — API импортируется из `vitest`).
- Тесты — `src/VirtualScroller.test.tsx`, покрытие компонента 100%.
- **Важный нюанс виртуализации:** jsdom **не считает layout** — `scrollTop`/высоты/`getBoundingClientRect` там фейковые. Поэтому проверяется *логика выбора индексов*: `scrollTop` задаётся вручную через `fireEvent.scroll(el, { target: { scrollTop } })`, а не реальным скроллом.
- Конфиг — `vitest.config.ts` (`environment: "jsdom"`, `setupFiles`, coverage v8 с исключением stories/index).

### E2E / визуал — Playwright (пока НЕ настроен)

Следующий логичный слой: **Playwright** гоняет реальный браузер — проверяет фактический скролл, offset'ы и визуальную регрессию (скриншот-диффы). Натравливается на собранный Ladle (`npm run build:ladle` → статика). Для этой либы оправдан сильнее обычного, т.к. скролл/layout — суть компонента.

Матрица: Vitest+RTL на каждый коммит (CI) → Ladle для ручного дева → Playwright точечно на реальный скролл/визуал.
